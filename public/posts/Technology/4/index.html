Authorization Technique

I. OAuth 2.0

1. OAuth란?

  • 액세스 위임을 위한 개방형 표준 인가 프로토콜
  • 타사 애플리케이션이 사용자 비밀번호를 공유하지 않고도 사용자 계정에 제한된 액세스를 허용할 수 있는 권한을 부여받는 방법
    • 로그인 절차 간소화
    • 보안 유지

2. OAuth 2.0 > OAuth 1.0

  • 단순해진 로직
    • 각 요청에 대해 서명을 하지 않음
  • 폭넓은 지원
    • 웹 기반 클라이언트를 넘어서 desktop, mobile, server2server까지 지원
  • 토큰
    • 토큰 유효기간이 짧고 종류가 다양
    • refresh token을 사용해 access token 갱신

3. Authorization Grant

  • Authorization Code Grant
    • 동작
      • 클라이언트가 사용자를 권한 부여 서버로 redirect해서 인증을 요청
      • 사용자가 로그인하고 권한을 부여하면, 권한 부여 서버는 클라이언트에게 Authorization Code(인가 코드)를 발급
      • 클라이언트는 인가 코드를 권한 부여 서버에 다시 보내고, 그 대가로 액세스 토큰을 받음
      • 액세스 토큰을 사용해 리소스 서버에 접근
    • 특징
      • 서버 기반 애플리케이션에 주로 사용
      • 클라이언트가 직접 사용자 자격 증명을 다루지 않고, 인가 코드를 통해 안전하게 액세스 토큰을 교환
      • 민감한 정보가 브라우저에 노출되지 않아 안전함
  • Implicit Grant
    • 동작
      • 클라이언트가 사용자를 권한 부여 서버로 redirect해서 권한 요청
      • 사용자가 로그인하고 권한을 부여하면, 권한 부여 서버는 클라이언트에게 직접 액세스 토큰을 발급
      • 클라이언트는 이 액세스 토큰을 사용해 리소스 서버에 접근
    • 특징
      • 클라이언트 측 애플리케이션, 특히 SPA(Single Page Application)와 같이 브라우저에서 실행되는 앱에서 주로 사용됨
      • 액세스 토큰이 직접 전달되며, 인가 코드를 교환하는 과정이 없는 대신 액세스 토큰이 노출되기 쉬워 보안이 낮음
  • Resource Owner Password Credentials Grant
    • 동작
      • 사용자가 클라이언트에게 직접 자신의 아이디와 비밀번호를 입력
      • 클라이언트는 이 정보를 권한 부여 서버로 보내고, 액세스 토큰을 요청
      • 권한 부여 서버는 사용자를 인증한 후 액세스 토큰을 발급
    • 특징
      • 인증 과정이 매우 단순하고 보안에 취약
      • 신뢰할 수 있는 애플리케이션에서만 사용 가능
  • Client Credentials Grant
    • 동작
      • 클라이언트는 자신의 클라이언트 아이디와 클라이언트 비밀키를 권한 부여 서버로 보내고, 액세스 토큰을 요청
      • 권한 부여 서버는 클라이언트 자격 증명을 확인한 후 액세스 토큰을 발급
      • 클라이언트는 이 액세스 토큰을 사용해 리소스 서버에 접근
    • 특징
      • 서버 간 통신 또는 머신 간 통신에 주로 사용
      • 사용자가 개입하지 않고, 클라이언트가 자신의 자격 증명으로 리소스 서버에 직접 접근할 때 사용 (ex 백엔드 서비스 사이의 API 호출)

II. SSO (Single Sign-On)

사용자가 한 번의 로그인으로 여러 애플리케이션이나 시스템에 접근할 수 있도록 하는 인증 방식

한 번 로그인하면 다른 애플리케이션에 대해서는 별도의 인증 절차 없이 자동으로 접근 허용

동작

  1. 첫 로그인
    • 사용자가 SSO 시스템이 적용된 애플리케이션 중 하나에 처음 접근하면 로그인 페이지로
  1. 인증 및 토큰 발급
    • 사용자가 자격 증명을 입력하면 중앙 인증 서버가 이를 확인, 사용자가 인증되면 토큰을 발급
      • 토큰 = 사용자의 인증 정보를 포함한 세션 정보
  1. 다른 서비스에 접근
    • 이후 사용자가 다른 애플리케이션에 접근할 때, 이미 발급된 인증 토큰이 전달되어 사용자가 추가로 로그인하지 않고도 해당 서비스에 접근 가능
    • 사용자가 로그아웃하거나 세션이 만료되지 않는 한 토큰은 유효함

특징

  • 한 번의 로그인을 여러 서비스에 쉽게 접근할 수 있어 편리함
  • 비밀번호를 여러 번 입력할 필요가 없으므로 노출될 위험이 감소함
    • 인증 정보가 유출되었을 때 피해가 커질 수 있음
  • 사용자 계정을 통합적으로 관리, 일관성 확보
    • 단일 실패 지점에 따른 문제점

프로토콜

  • SAML
  • OAuth, OpenID Connect
  • Kerberos

III. PKCE (Proof Key for Code Exchange)

OAuth 2.0에서 인가 코드 가로채기 공격을 방지하는 보안 확장 기능으로, 주로 모바일 앱이나 SPA(Single Page Application) 같은 공개 클라이언트에서 사용

동작

  1. Authorization Request (인가 요청)
    • 클라이언트는 랜덤한 문자열인 code verifier를 생성
    • code verifier를 해시(SHA-256)한 code challenge를 생성
    • 인가 요청을 할 때 code challenge를 포함해 권한 부여 서버로 보냄
  1. Authorization Response (인가 응답)
    • 사용자가 로그인하고 권한을 승인하면, 권한 부여 서버는 인가 코드를 클라이언트에게 발급
    • 권한 부여 서버는 code challenge를 저장
  1. Token Request (토큰 요청)

    클라이언트가 인가 코드를 받으면, 인가 코드를 사용해 액세스 토큰을 요청할 때, 처음에 생성한 code verifier를 함께 전송

  1. Token Response (토큰 응답)
    • 권한 부여 서버는 받은 code verifier를 해시하여 code challenge와 비교
    • 두 값이 일치하면 클라이언트를 신뢰, 액세스 토큰을 발급

특징

  • 인가 코드가 유출되어도 code verifier를 알지 못하면 액세스 토큰을 발급받을 수 없음
    • 보안상 더 안전함
