Git 명령어

git add index.htm : index.htm을 working directory에서 stage로 옮김 (untracked ⇒ tracked)

git add . : 전체 파일을 추적 상태로 등록

git status : 커밋할 내용이 있는지, 수정된 파일이 있는지 등, 깃의 상태 메시지를 출력

mkdir gitstudy01

cd gitstudy01

git init : 저장소를 git으로 초기화

cat index.htm

code index.htm

rm index.htm : index.htm 삭제

rm -f index.htm : index.htm 강제로 삭제

git mv index.htm home.htm : index.htm의 이름을 home.htm으로 바꿈

git rm --cached index.htm : 한번도 커밋하지 않은 stage 영역에 있는 파일을 unstage함

git commit

git commit -m "first commit"

git commit -am "second commit"

git commit --allow-empty-message -m ""

git commit -v : 이전과 달라진 영역의 내용을 ‘보면서(감상하면서)’ 커밋 메시지를 입력가능

git log

git log --graph --all

git log index.htm : index.htm에 관한 로그만 출력

git log --oneline : 7자리 커밋해시와 커밋 메시지를 묶어 한 줄씩 표현하여 출력

git log --graph --oneline : 위의 사항을 그래프로 도식화하여 출력

git diff : stage 영역과 working directory를 비교함

git diff head : 최신 커밋과 변경 내용을 비교

git remote add origin https://github.com/suhan8984/gitstudy01.git

git remote -v : 원격 저장소 목록 확인(fetch, push 주소를 출력함)

git remote show origin : 보다 상세한 원격 저장소 정보를 출력

git remote rm origin : 원격 저장소 삭제

git remote rename original origin : 원격 저장소 별칭을 original에서 origin으로 바꿈

git push origin master : master 브랜치를 원격 저장소에 전송함

git clone https://github.com/suhan8984/gitstudy02.git

: 저장소를 복제하여 서버에 있는 모든 커밋 이력을 내려받음

단, 트래킹 브랜치만 가져옴

git pull : 서버에서 갱신된 새 커밋을 내려받음

git fetch : 커밋을 가지고 오긴 하지만 로컬 저장소에 반영하지 않음

git merge origin/master 를 이용하여 로컬 저장소에 반영

  • 협업할 때에는 pull ⇒ coding ⇒ commit ⇒ pull ⇒ push 순서를 권장

git branch : branch 이름들의 목록을 보여줌 (현재 브랜치에 하이라이트 표시됨)

git branch -v : 각 브랜치의 상세 정보 출력(이름, 가장 최근 커밋 해시, 해당 커밋 메시지)

git branch footer : footer 브랜치 생성

git rev-parse footer : footer 브렌치의 head(가장 최근 말단)가 위치한 커밋 해시 값 출력

git checkout footer : footer 브랜치로 이동(워킹 디렉토리도 같이 바뀜)

git checkout -b footer : footer 브랜치를 생섬함과 동시에 footer 브랜치로 이동

git checkout dcdb1c1 : dcdb1c1을 해시키로 갖는 커밋으로 이동

git checkout HEAD~1 : Head의 바로 전 커밋으로 이동

git checkout - : 이전 커밋(브랜치)로 이동

git push -u origin hotfix : hotfix를 원격 서버로 전달

git push -u origin feature:function

⇒ feature 브랜치를 function 브랜치로 이름을 바꿔 푸시

upstream tracking 정보를 생성하므로 -u를 쓰는 것을 권장

git branch -r : 리모트 브랜치 목록을 출력

git branch -vv : 트래킹 브랜치 목록을 출력

git checkout --track origin/function : 업스트림 브랜치 function을 생성 (원격 저장소에 function을 tracking하는 로컬 저장소의 function 브랜치를 만들어 그곳으로 이동)

git fetch, git checkout -b aaa origin/aaa : 원격 저장소에서 aaa를 origin/aaa로 fetch를 통해 받아온 다음 로컬저장소에 aaa 브랜치를 만들어 origin/aaa를 적용하고 그 브랜치로 checkout한다.

git branch -u origin/bbb : 현재 브랜치를 원격 저장소의 브랜치 bbb와 업스트림 설정

git push --set-upstream origin master : 현재 master 브랜치를 origin 서버의 master로 업스트림 설정

git branch -d bug : bug 브랜치를 삭제(스테이지 상태가 깨끗해야 삭제 가능)

git branch -D hotfix : hotfix 브랜치를 강제로 삭제

git push origin --delete aaa : 원격 저장소의 브랜치 aaa를 삭제함

git stash : 작업 중인 내역을 스택에 저장함

git stash list : 스태시에 저장된 스택을 확인

git stash show : 방금 전의 스태시와 현재 워킹 디렉터리 간 차이를 출력

git stash show -p stash@{0} : 스택 0번지에 저장된 스태시와 현재 워킹 디렉터리간 차이를 상세하게 출력

git stash pop : 스태시의 스택에 저장된 작업 내용이 적용(병합)됨. 병합에 성공하면 스택에서 제거

git stash apply stash@{1} : apply는 스택 목록의 중간 작업을 지정하여 적용할 수 있음. 병합에 성공해도 스택에서 제거하지 않음

git stash drop : 가장 최근의 스태시 삭제

git clean : untracked 상태의 파일을 삭제

git clean -f : 강제로 untracked 파일 삭제

(master) git merge feature : feature 브랜치를 master에 병합

git reset --hard HEAD^ : 직전에 병합을 한 경우 이를 취소

git merge hotfix --edit : 병합 커밋 메시지를 직접 작성하려고 하는 경우 --edit 추가

git branch -d hotfix : 병합 후 필요없어진 브랜치 hotfix를 삭제

git merge --abort : 병합하려고 했는데 충돌이 일어나서 병합을 취소하고 싶을 때

⇒ 충돌이 일어난 경우 수동으로 직접 해결하고 커밋하면 됨

git branch --merged : 병합한 브랜치 출력

git branch --no-merged : 병합하지 않은 브랜치 출력

(description) git rebase master : 파생 브랜치에서 메인 브랜치를 rebase해야함 파생 브랜치의 head가 메인보다 앞으로 나가있어서 fast-forward 병합필요

git checkout master

git merge description : fast-forward 방식으로 병합 진행

충돌이 발생한 경우 수동으로 직접 수정한다음 stage등록, git rebase --continue를 이용해서 충돌된 부분들을 한 단계씩 해결해 나가면서 병합

마찬가지로 필요없어진 description 브랜치는 제거

git rebase --abort : 직전 rebase 취소

git rebase -i HEAD~3 를 한 다음 두 번째부터 마지막 줄의 pick를 squash로 바꾸면 커밋 3개를 하나의 커밋으로 묶을 수 있음. 커밋 해시값이 바뀜

⇒ 저장소를 외부에 공개했다면 리베이스는 하지 않는 것이 원칙

git reset --soft HEAD~2 : 1, 2, 3, 4, 5 순으로 커밋을 했다면 3의 커밋지점으로 돌아가고 스테이지 영역에는 4, 5 또한 수행된 파일이 올라가며, 워킹디렉터리를 수정하지는 않기 때문에 5를 커밋한 이후에 진행된 수정사항은 워킹디렉터리상에 보존이 된다. (파일 자체는 수정사항이 그대로라는 뜻)

git reset --mixed HEAD~2 (--mixed가 기본값이므로 생략해도 무방) :

1, 2, 3, 4, 5 순으로 커밋을 했다면 3의 커밋지점으로 돌아가고 스테이지 영역은 비어있고, 워킹디렉터리는 그대로이다.

git reset --hard HEAD~2 : 1, 2, 3, 4, 5 순으로 커밋을 했다면 3의 커밋지점으로 돌아가고 스테이지 영역은 비어있고, 워킹디렉터리도 3을 커밋한 시점으로 돌아간다.

  1. ex) git reset --soft HEAD~2 git commit -am "최근 3개 커밋을 합침"

git reset menu.htm : menu.htm에 대한 stage 상태를 제거(unstage)

git reset --hard HEAD : 워킹 디렉터리와 스테이지 상태를 모두 제거하여 마지막 커밋 상태로 되돌림

git reset --merge HEAD~ : 직전에 병합한 커밋을 리셋하여 취소함

⇒ 협업하는 경우에는 reset을 하지 않음

git revert HEAD : 현재 커밋을 취소하는 커밋 생성

git revert --mainline 1 84b6618 : 84b6618 커밋에서 병합이 이루어졌다고 가정함. 파생 브랜치와 병합됨에 의해 변경된 사항을 취소함.

git tag : 태그의 목록을 출력

git tag -a 1.0.0 : HEAD에 annotated tag 1.0.0을 붙임

git tag -a 1.1.0 -m "simple tag 1.1.0" : annotated tag message까지 같이 입력함

git tag -d 1.1.0 : 1.1.0 태그를 삭제함

git show 1.0.0 : 해당 태그와 그 커밋 정보를 출력

git tag 2.0.0 : lightweight tag 2.0.0을 생성. 태그 메시지는 입력할 수 없음.

git tag 1.1.2 80f8890 : 커밋 아이디가 80f8890인 커밋에 1.1.2 태그를 붙임

git checkout 1.0.0 : 1.0.0 태그가 붙은 커밋으로 체크아웃

git checkout -b fix 1.0.0 : 1.0.0 태그가 붙은 커밋을 기준으로 fix 브랜치를 생성한 후 체크아웃함

git push origin 1.0.0 : 원격저장소에 1.0.0 태그를 전송 (push만으로는 태그전송이 안됨)

git push origin --tags : 원격저장소에 모든 태그를 전송

git push --delete origin 2.0.0 : 원격저장소에서 2.0.0 태그를 삭제

git push origin 1.1.2:3.0.0 ⇒ 로컬 태그 1.1.2를 원격 저장소 태그 3.0.0로 전송

git submodule add https://github.com/suhan8984/child.git child

: 원격저장소 child.git을 서브 저장소로 설정하여 폴더명이 child인 서브 폴더 형태로 취급

git add .gitmodules

git commit -m "add submodule"

서브모듈에서 커밋할 사항이 있으면 child, parent 순서로 커밋해야함

local 서브모듈 child에서 수정사항을 커밋해도 local child 원본과 remote child 모두 변하지 않는다. 세 child는 기본적으로 독립적이고, push, pull에 의해서 서로 진행상황을 업데이트할 뿐이다.

git submodule update : 부모저장소을 clone하거나 pull 한다고 해서 서브모듈의 정보를 가져올 수 있는 것은 아님. 따라서 서브모듈을 업데이트하도록 명령해야함.

git show d0943cf : 해당 해시값에 해당하는 커밋 로그를 출력

git rev-parse master : master가 참조하는 해시값(Head의 해쉬값)을 조회

git reflog : 참조 기록 확인

git blame index.htm : index.htm 파일을 수정한 커밋에 대하여 해시값, 작성자, 수정한 코드, 수정한 시간 등을 출력함

git blame -L 34, 56 index.htm : index.htm의 34~56번째 줄을 수정한 커밋 히스토리 출력

git replace c3502a6 8681d97 : 두 커밋을 연결함 (두 커밋 해시값이 동일한 것으로 간주)

git prune --dry-run --verbose : 고립된 객체를 삭제하려고 하는데 실행하지는 않고 작업할 내역만 출력

git reflog expire --expire=now --expire-unreachable=now --all

: 고립될만한 객체를 실제로 고립시키기 위해 reflog 기록에서 연결 고리 링크를 제거

git prune --expire now -v : 고립 객체 삭제

git config rerere.enabled true : rerere를 활성화

⇒ rerere는 이전의 충돌 해결 작업을 기억하고 이후에 유사한 문제가 발생하면 이를 다시 적용함

git rerere status : 충돌을 일으킨 파일의 목록을 출력

git rerere diff : 충돌한 내용의 차이점을 출력
